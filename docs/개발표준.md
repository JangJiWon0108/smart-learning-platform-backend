# 백엔드 개발 표준 (Backend Development Standards)

`smart-learning-platform-backend` 프로젝트의 코드 일관성과 유지보수성 향상을 위한 가이드라인.

---

## 1. 파일 명명 규칙
- **LLM 에이전트**: `{역할}_agent.py` (예: `solver_agent.py`)
- **전처리 노드**: `{역할}_nodes.py` (예: `solver_nodes.py`)
- **스키마**: `{역할}_output.py` (예: `solver_output.py`)
- **설정 파일**: `properties.py`
- **임포트 순서**: 1. `__future__` 2. 표준 라이브러리 3. 서드파티 4. 로컬 모듈(절대경로)
- **접근 제한**: 공개 API(접두사 없음), 내부 헬퍼 함수(`_`), 모듈 상수(`_UPPER_SNAKE_CASE`)

---

## 2. 주석 및 독스트링 (Docstring)

### 2.1 파일 및 모듈
- 모든 `.py` 파일 최상단에 해당 모듈의 역할과 핵심 로직을 설명하는 한글 독스트링 작성 필수.

### 2.2 함수 및 클래스 (Google 스타일)
- 모든 인자와 반환값에 대한 설명 기재.
- ADK 노드 함수의 경우 `state에 저장되는 값:` 항목 명시 필수.
```python
def node_func(param1: str) -> None:
    """
    {요약 및 역할 설명}
    Args: param1 설명
    Yields: Event
    state에 저장되는 값:
        - key_name: 설명
    """
```

### 2.3 논리적 블록 구분 및 인라인 주석
- 연관된 코드 뭉치(설정, 헬퍼, 실행부 등)는 시각적 구분선(`# ─── 파트명 ───`)으로 분리.
- 주요 모듈 변수 위에 목적 기재 및 복잡한 로직의 단계별 주석(`# 1단계: ...`) 추가.

---

## 3. 타입 시스템 및 스키마

### 3.1 타입 힌트
- 파라미터 및 반환값 타입 명시 필수 (Python 3.10+ `X | None` 문법 권장).

### 3.2 Pydantic 스키마 (`schemas/`)
- 필수 필드는 `Field(...)`, 선택 필드는 `Field(default=...)` 사용.
- 모든 필드에 `description` 작성 및 선택지 제한 시 `Literal` 활용.

---

## 4. Google ADK 구현 가이드

프로젝트 핵심 프레임워크인 ADK 활용 시의 반복 패턴 준수.

### 4.1 에이전트 정의 (`llm_agents/`)
```python
agent = Agent(
    name="intent_agent",  
    model=settings.GEMINI_MODEL_TYPE_INTENT, # 용도별 모델 설정 변수 참조
    generate_content_config=GEMINI_GENERATE_CONTENT_RETRY_CONFIG, # 공통 API 설정 재사용
    output_schema=IntentOutput, # Pydantic 모델 지정
    output_key="intent_result", # state 저장 키 이름
    after_agent_callback=intent_callback_func, # 실행 후 콜백
    tools=[], # 필요 시 도구 목록
    description="사용자 질문 의도 분류 에이전트",
    instruction=_INSTRUCTION # 모듈 상단 상단 정의 지시문
)
```

### 4.2 주요 파라미터 규칙
- **name**: 영문 소문자와 언더스코어 조합의 고유 식별자 사용.
- **model**: `settings` 상수를 통한 모델 지정 (직접 문자열 입력 금지).
- **generate_content_config**: 온도, 재시도 등 공통 설정 파일 참조.
- **output_schema**: LLM 응답 구조화 스키마. 모든 `Field`에 구체적인 `description` 작성 필수.
- **output_key**: 워크플로우 `state`에 결과가 저장될 키 이름 정의.
- **after_agent_callback**: 데이터 재가공 및 상태 조합을 위한 콜백 패턴 활용. `callback_context.state`를 통한 상태 접근 및 가공 후 `state["key"]` 업데이트.
- **tools**: 검색, 외부 API 연동 등 모델이 직접 호출할 함수 목록.
- **description**: 에이전트 역할 요약 (로깅 및 관리 목적).
- **instruction**: 에이전트 페르소나 및 동작 규칙 정의. 모듈 최상단 `_INSTRUCTION` 상수로 분리 관리.
    1. 역할 선언 (`당신은 ... 입니다`)
    2. state 변수 참조 (`{state_key}` 형식)
    3. 규칙 섹션 구분 (마크다운 헤딩 활용)
    4. 출력 형식 강제 (JSON 응답 시 마크다운 코드 블록 제외 지시)

### 4.3 노드 및 라우팅 (`nodes/`)
- LLM 호출 없는 데이터 처리 및 흐름 분기 로직.
- 파라미터 이름과 ADK state 키 이름을 일치시켜 자동 주입 활용.
- 결과값은 `yield Event(state={"key": value})` 형태로 반환하여 글로벌 상태 업데이트.

---

## 5. 기타 컨벤션

- **에러 핸들링**: API 레이어에서 `HTTPException` 및 한글 메시지 사용. 입력값 방어적 검사 필수.
- **로깅**: `get_logger` 활용. `info`, `warning` 등 적절한 레벨로 기록.
- **비동기 처리**: SSE 스트리밍 시 `AsyncGenerator` 및 `async for` 패턴 준수.
